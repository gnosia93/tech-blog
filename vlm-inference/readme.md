## AWS GPU 인스턴스로 1B VLM 추론하기 ##

멀티모달 LLM이 빠르게 작아지고 있습니다. 한때 수십억 파라미터가 기본이던 비전-언어 모델(VLM, Vision-Language Model)이 이제는 1B(10억) 규모로도 이미지와 영상을 꽤 잘 이해합니다. 덕분에 "굳이 A100 여러 장을 붙이지 않아도 영상 이해 서비스를 만들 수 있지 않을까?"라는 질문이 현실적인 검토 대상이 되었습니다.

이 글에서는 AWS 솔루션즈 아키텍트 관점에서 세 가지를 정리합니다. 첫째, 영상 데이터가 실제로 모델에 어떻게 전달되는가. 둘째, 그 과정에서 GPU 메모리를 결정하는 진짜 변수는 무엇인가. 셋째, 그래서 어떤 EC2 인스턴스를 골라야 하는가.

결론부터 말하면, 1B 멀티모달 모델의 GPU 메모리는 **모델 가중치가 아니라 입력 영상이 만들어내는 토큰 수**가 좌우합니다. 이 한 문장이 이 글 전체를 관통하는 핵심입니다.

## 1. 먼저 용어부터: 이건 "동영상 모델"이 아니라 "VLM 인퍼런스"

본론에 들어가기 전에 용어를 정리하는 게 좋겠습니다. "동영상 모델 인퍼런스"라는 표현은 흔히 쓰이지만 실무에서는 오해를 부릅니다. 영상을 *생성*하는 Sora 같은 Video Generation, 영상을 분류하는 Action Recognition, 영상을 압축하는 Codec까지 전부 "동영상 모델"로 묶여버리기 때문입니다.

우리가 다루는 것은 **영상을 입력받아 이해하고 언어로 답하는** 작업입니다. 이 분야의 정확한 용어는 **VLM(Vision-Language Model) 인퍼런스** 또는 **Video-LLM 인퍼런스**이고, 태스크 관점에서는 Video Understanding / Video Question Answering이라고 부릅니다. 자료를 찾을 때도 이 키워드를 쓰는 편이 정확합니다.

## 2. 영상은 모델에 어떻게 전달될까

가장 먼저 짚어야 할 오해가 있습니다. 영상은 mp4 파일 그대로 모델에 들어가지 않습니다. 영상은 **프레임 단위 이미지로 쪼개진 뒤, 각 프레임이 이미지 토큰으로 변환되고, 텍스트 토큰과 하나의 시퀀스로 합쳐져** LLM에 입력됩니다.

전체 흐름은 다음과 같습니다.

```
   ┌──────────────────────────────────────────────────────────┐
   │                    동영상 파일 (mp4 등)                     │
   └──────────────────────────────────────────────────────────┘
                              │
             ① 디코딩 & 프레임 추출 (샘플링)  ← N을 여기서 결정
                              ▼
        ┌────┐ ┌────┐ ┌────┐ ┌────┐        ┌────┐
        │ f1 │ │ f2 │ │ f3 │ │ f4 │  ····  │ fN │   프레임 이미지
        └────┘ └────┘ └────┘ └────┘        └────┘
                              │
             ② 전처리 (resize / normalize)
                              ▼
             ③ 비전 인코더 (ViT) → 패치 임베딩
                              ▼
             ④ 프로젝터(MLP) + 토큰 압축  ← 메모리 절약 핵심
                              ▼
   ┌──────────────────────────────────────────────────────────┐
   │ [시스템] <video> f1_tok f2_tok ... fN_tok </video> [질문]  │
   └──────────────────────────────────────────────────────────┘
                              │
                         ⑤ LLM (1B)
                              ▼
                            출력
```

각 단계를 짧게 짚어보겠습니다.

**① 프레임 추출(샘플링)** 이 단계가 뒤에 나올 모든 메모리 계산의 출발점입니다. 30fps 영상은 1분에 1,800프레임이지만, 이걸 다 넣지 않습니다. 균등 샘플링(전체에서 N장 고르게)이나 FPS 기반 샘플링(초당 1프레임 등)으로 **N장만** 골라냅니다. 이 N이 곧 토큰 수와 VRAM을 직접 결정합니다.

**② 전처리** 각 프레임을 모델이 요구하는 해상도로 resize하고 정규화합니다. 해상도가 높을수록 프레임당 토큰이 늘어납니다(예: 336×336 → 576토큰).

**③ 비전 인코더** 각 프레임을 패치(예: 14×14)로 나눠 ViT가 임베딩으로 바꿉니다. 프레임 한 장이 수백 개의 비주얼 토큰이 됩니다.

**④ 프로젝터 정렬** 비전 임베딩을 LLM이 이해하는 차원으로 매핑합니다. 많은 모델이 바로 이 지점에서 토큰 압축(merging/pooling)을 수행해 프레임당 토큰을 줄입니다. 메모리 절약의 핵심 레버가 여기 있습니다.

**⑤ 텍스트와 결합** 최종 입력은 `[시스템 프롬프트] <video> 프레임 토큰들 </video> [질문]` 형태의 단일 시퀀스가 됩니다.

실제 API 호출은 두 가지 방식으로 나뉩니다. 하나는 서버가 영상을 직접 디코딩하는 방식이고,

```python
# 서버가 영상을 직접 디코딩 (예: Qwen2-VL 스타일)
messages = [{
    "role": "user",
    "content": [
        {"type": "video", "video": "file:///path/video.mp4", "fps": 1.0},
        {"type": "text", "text": "이 영상에서 무슨 일이 일어나?"}
    ]
}]
```

다른 하나는 클라이언트가 미리 프레임을 뽑아 이미지 리스트로 보내는 방식입니다.

```python
# 클라이언트가 프레임을 미리 추출해 이미지 리스트로 전달
content = [
    {"type": "image", "image": frame1},
    {"type": "image", "image": frame2},
    # ... N장
    {"type": "text", "text": "이 프레임들을 순서대로 보고 설명해줘"}
]
```

## 3. GPU 메모리를 결정하는 진짜 변수

이제 핵심입니다. 1B 모델이라고 하면 흔히 "가중치 2GB니까 8GB GPU면 충분하겠지"라고 생각하기 쉽지만, 영상에서는 이야기가 달라집니다.

메모리는 크게 두 덩어리로 나뉩니다.

**첫째, 모델 가중치.** 이건 예측 가능하고 고정적입니다.

```
   정밀도별 1B 가중치
   FP16/BF16  ██████████        약 2 GB
   INT8       █████             약 1 GB
   INT4       ██                약 0.5 GB
```

여기서 한 가지 주의할 점은 "1B"가 LLM만 가리키는지, 비전 인코더까지 포함하는지입니다. 보통 비전 인코더(ViT)가 0.3~0.4B 정도 추가로 붙습니다.

**둘째, KV 캐시와 액티베이션.** 이게 변동성이 크고, 영상에서 폭발하는 부분입니다. 멀티모달 모델은 이미지·영상을 토큰으로 바꾸는데, 이 **총 토큰 수**가 KV 캐시 크기를 좌우합니다.

이미지 한 장은 보통 256~1,024토큰이라 부담이 적습니다. 그런데 영상은 `프레임 수 × 프레임당 토큰`이라, 프레임당 256토큰짜리를 32프레임 넣으면 8,000토큰을 훌쩍 넘깁니다.

1B급(레이어 약 24, hidden 약 2,048) 기준으로 FP16 KV 캐시는 대략 토큰당 200KB 내외입니다. 이걸 그림으로 보면 토큰 수와 메모리가 선형으로 붙어 있다는 게 분명해집니다.

```
   총 비주얼 토큰 수  →  KV 캐시 (FP16, 토큰당 ~200KB)

     8,000 토큰   ███                   약 1.5 GB
    16,000 토큰   ██████                약 3 GB
    32,000 토큰   ████████████          약 6 GB+   ← 긴 영상
```

즉 VRAM을 줄이는 레버는 세 가지로 정리됩니다. 프레임 수를 줄이거나(64 → 16장), 해상도를 낮추거나, 애초에 토큰 압축이 강한 모델을 고르는 것입니다.

## 4. 같은 1B라도 아키텍처마다 토큰 처리가 다르다

"1B니까 어느 모델이든 비슷하겠지"는 영상에서 특히 위험한 가정입니다. 모델마다 **프레임 한 장을 몇 개의 토큰으로 바꾸는지**가 다르고, 그 차이가 그대로 VRAM 차이로 이어지기 때문입니다.

핵심은 토큰 압축 기법입니다. 크게 세 갈래로 나뉩니다.

```
   ┌─────────────────┬──────────────────────────────┬──────────────────┐
   │ 기법            │ 동작                          │ 대표 모델         │
   ├─────────────────┼──────────────────────────────┼──────────────────┤
   │ Pixel Shuffle   │ 인접 패치를 채널로 접어        │ InternVL,        │
   │ (space-to-depth)│ 토큰을 1/4 ~ 1/16로 압축      │ SmolVLM          │
   ├─────────────────┼──────────────────────────────┼──────────────────┤
   │ Patch Merging   │ 2×2 인접 패치를 하나로 병합    │ Qwen2-VL         │
   │                 │ (4배 감소)                    │                  │
   ├─────────────────┼──────────────────────────────┼──────────────────┤
   │ Pooling / 그대로│ 압축 거의 없이 ViT 패치 사용   │ LLaVA-1.5        │
   └─────────────────┴──────────────────────────────┴──────────────────┘
```

**LLaVA(특히 1.5)**는 336×336 입력을 24×24 = 576토큰으로 고정 변환하며 압축이 거의 없습니다. 구조가 단순하고 생태계 호환성이 좋지만, 영상에서는 프레임당 토큰이 많아 메모리 부담이 큽니다.

**Qwen2-VL**은 해상도에 비례해 토큰이 변하는 Naive Dynamic Resolution을 쓰고, 2×2 patch merging으로 4배 압축합니다. 무엇보다 `min_pixels`/`max_pixels`로 토큰 상한을 직접 제어할 수 있어 서빙 시 메모리 예측과 튜닝이 쉽습니다. 영상은 시간축을 포함한 3D 위치 인코딩으로 비교적 효율적으로 처리합니다.

**InternVL**은 이미지를 종횡비에 맞춰 여러 448×448 타일로 쪼개고(예: 최대 12타일 + 썸네일), 각 타일을 pixel shuffle로 약 256토큰까지 압축합니다. 고해상도 문서나 디테일 인식에 강하지만, 타일 수를 제한하지 않으면 토큰이 급증합니다.

**SmolVLM(영상은 SmolVLM2)**은 애초에 극단적 압축을 목표로 설계됐습니다. aggressive pixel shuffle로 프레임당 토큰을 수십~81 수준까지 낮춰, 엣지나 소형 GPU 환경을 노립니다. 대신 세밀한 디테일은 다소 손해를 봅니다.

이 차이를 "16프레임 영상"이라는 동일 조건에 대입하면 VRAM 차이가 뚜렷하게 드러납니다.

```
   모델              프레임당 토큰   16프레임 총 토큰    상대 VRAM
   ─────────────────────────────────────────────────────────────
   SmolVLM           64 ~ 81         1,000 ~ 1,300      최소   ▏
   Qwen2-VL(제한 시)  수백(조절가능)  조절 가능           낮음~중간 ▎
   InternVL(1~2타일)  256 ~ 512       4,000 ~ 8,000      중간   ▍▍
   LLaVA-1.5          ~576            ~9,000             높음   ▍▍▍▍
```

토큰당 200KB(FP16) 기준으로 환산하면, SmolVLM의 1,300토큰은 약 0.25GB, LLaVA의 9,000토큰은 약 1.7GB입니다. 긴 영상으로 토큰이 3만을 넘으면 6GB 이상으로 뛰고요. 같은 영상, 같은 1B급인데도 어떤 압축 방식을 쓰느냐에 따라 VRAM이 수 배 차이 나는 셈입니다.

## 5. AWS GPU 인스턴스 라인업 한눈에 보기

이제 실제 인프라입니다. GPU를 고르기 전에, AWS가 제공하는 GPU 인스턴스 패밀리를 먼저 훑어보겠습니다. 크게 **추론·그래픽에 최적화된 G 계열**과 **대규모 학습·초대형 추론용 P 계열**로 나뉩니다. 1B급 VLM 영상 인퍼런스라면 대부분 G 계열 안에서 답이 나옵니다.

```
   추론 지향 (G 계열)                        학습/초대형 (P 계열)
   ─────────────────────────────           ─────────────────────────
   G4dn   T4    16GB   가장 저렴            P4d   A100  40GB
   G5     A10G  24GB   추론 표준            P4de  A100  80GB
   G6     L4    24GB   최신 추론 (가성비)    P5    H100  80GB
   G6e    L40S  48GB   대용량/영상 스케일업  P5en  H200  141GB
   ────────────────────────────────────────────────────────────────
      비용 낮음  ◀─────────────────────────────────▶  성능 높음
```

패밀리별로 정리하면 다음과 같습니다.

| 패밀리 | GPU | GPU당 VRAM | 성격 | 1B VLM 영상 인퍼런스 관점 |
|--------|-----|-----------|------|--------------------------|
| **G4dn** | NVIDIA T4 | 16GB | 가장 저렴한 추론용 | SmolVLM 등 경량·저프레임에 적합 |
| **G5** | NVIDIA A10G | 24GB | 추론의 표준 워크호스 | 무난한 시작점, 검증된 선택 |
| **G6** | NVIDIA L4 | 24GB | 최신 세대, 가성비 우수 | **기본 추천** (L4 신세대) |
| **G6e** | NVIDIA L40S | 48GB | 대용량 메모리 | 긴 영상·다중 동시요청 스케일업 |
| **P4d / P4de** | NVIDIA A100 | 40 / 80GB | 대규모 학습·고처리량 | 대규모 배치 서빙 시 |
| **P5 / P5en** | NVIDIA H100 / H200 | 80 / 141GB | 최상위 성능 | 1B급엔 과잉, 초대형 모델용 |

G6e는 인스턴스당 최대 8장의 L40S(총 384GB)까지 확장되며, 단일 GPU만 필요한 1B급이라면 `g6e.xlarge`(L40S 1장, 48GB)부터 시작할 수 있습니다. 각 수치는 AWS 공식 사양 기준이며, 리전·가용성은 배포 전 확인하시길 권합니다.

## 6. 그래서 어떤 EC2 인스턴스를 골라야 하나

앞서 계산한 대로 1B 멀티모달 + 영상 인퍼런스의 무난한 시작점은 **24GB GPU 한 장**입니다. 가중치 3GB(LLM + 비전 인코더)에 영상 KV 캐시 1~6GB, 그리고 액티베이션과 프레임워크 오버헤드까지 얹어도 24GB면 여유가 있습니다. 이걸 위 라인업에 대입하면 자연스럽게 **G6(L4 24GB)**가 기본 후보가 됩니다.

의사결정을 흐름으로 정리하면 이렇습니다.

```
                 영상 인퍼런스가 목표?
                        │
              ┌─────────┴─────────┐
             예                   아니오(이미지만 / 경량 모델)
              │                        │
       동시요청 많거나           G4dn(T4 16GB)
       긴 영상(64f+)?          또는 G5/G6 소형
              │
      ┌───────┴────────┐
     예                아니오
      │                  │
  g6e.xlarge         g6.xlarge (L4 24GB)  ◀ 기본 추천
  (L40S 48GB)        또는 g5.xlarge (A10G 24GB)
  또는 p4d(A100)
```

정리하면 이렇습니다. 대부분의 팀은 **g6.xlarge(L4 24GB) 한 장 + vLLM**으로 시작하면 됩니다. 최신 세대라 가성비가 좋고, vLLM의 PagedAttention 덕분에 KV 캐시 관리까지 효율적입니다. 검증된 안정성을 원한다면 **g5.xlarge(A10G 24GB)**도 훌륭한 대안입니다.

트래픽이 낮고 SmolVLM처럼 극단적으로 가벼운 모델을 쓴다면 **G4dn(T4 16GB)**으로 비용을 더 낮출 수 있고, 반대로 긴 영상이나 높은 동시 처리량이 필요해지면 **G6e(L40S 48GB)**, 그 이상은 **P4d(A100)**로 스케일업하는 경로가 자연스럽습니다.

## 마치며

1B 멀티모달 모델의 GPU 산정에서 가장 흔한 실수는 "작은 모델이니 작은 GPU면 된다"고 가중치만 보는 것입니다. 실제로 VRAM을 좌우하는 건 **영상이 만들어내는 토큰 수**이고, 그 토큰 수는 프레임 샘플링 개수와 모델의 토큰 압축 방식에서 결정됩니다.

그래서 GPU 선택은 사실 세 가지 질문으로 환원됩니다. 몇 프레임을 넣을 것인가, 어떤 압축 방식의 모델을 쓸 것인가, 동시 요청은 얼마나 되는가. 이 세 가지가 잡히면 필요한 VRAM이 계산되고, 거기에 맞는 EC2 인스턴스가 자연스럽게 정해집니다. 그리고 대부분의 출발점은 **G6(L4 24GB) 한 장 + vLLM**입니다.



*본문의 토큰 수치는 대표 설정 기준 근사치이며, 발행 전 각 모델의 프로세서 config로 확인을 권합니다. AWS 인스턴스 사양은 [G6e](https://aws.amazon.com/ec2/instance-types/g6e/), [G5](https://aws.amazon.com/ec2/instance-types/g5/) 등 공식 페이지 기준이며, 리전·가용성·가격은 시점에 따라 달라질 수 있습니다.*
